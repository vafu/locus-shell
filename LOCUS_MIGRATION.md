# Typed Locus Model And GTK Binding Migration

## Summary

First update `~/proj/locus` so the YAML schema can describe typed graph properties and generate Rust contracts: models, property accessors, and common paths. Then update `locus-shell` to consume those generated contracts through `locus-provider`, a small graph integration crate that owns the generated binding type and provider implementation, while macros bind typed Locus fields into Relm4 model state.

## Phase 1: Type The Locus Schema

- Extend `locus-schema` with a `PropertyType` enum:
  - `string`
  - `bool`
  - `u32`
  - `i32`
  - possibly `f64`
- Update YAML parsing so property specs can be either shorthand or typed:
  - `name: {}` defaults to `string`
  - `active: { type: bool }`
  - `index: { type: u32 }`
  - `required: true` continues to work
- Update schema validation tests for typed property parsing.
- Keep runtime graph storage as strings for now; type information is for validation/codegen first.

## Phase 2: Add Typed Property Validation

- Decide whether `locusd` rejects invalid property values at write time.
- Recommended: yes, once property types exist.
- Add validation in `locus-core` before accepting `SetProperty`.
- Parse string values according to schema type:
  - `bool`: `true` / `false`
  - integers: normal decimal parsing
  - `string`: any string
- Keep D-Bus wire format unchanged initially: properties remain strings.
- Add tests for accepted/rejected typed writes.

## Phase 3: Generate Rust Contracts From Locus

- Add `--language rust` to `locus-codegen`.
- Generate a Rust module or crate containing:
  - node model structs/newtypes: `Window`, `Workspace`, `Project`, etc.
  - property descriptors: `Workspace::id`, `Workspace::focused`, `Project::name`
  - named path descriptors: `paths::selected_workspace`, `paths::selected_project`
  - typed binding descriptors combining source/path/property
- Generated property descriptors should carry:
  - property key
  - Rust value type
  - decode function from D-Bus string
- Generated paths should carry:
  - source
  - relation path
  - whether the path is one or many
  - target model kind when known
- Add generated-code snapshot tests or golden-file tests.

## Phase 4: Define The Rust Contract Shape

Target generated API should support expressions like:

```rust
paths::selected_workspace().property(Workspace::id)
paths::selected_project().property(Project::name)
```

or equivalent associated constants:

```rust
paths::SELECTED_WORKSPACE.property(Workspace::ID)
paths::SELECTED_PROJECT.property(Project::NAME)
```

The expression should produce a typed `FieldBinding<T>`:

```rust
FieldBinding<u32>
FieldBinding<String>
FieldBinding<bool>
```

This type should be transport-neutral enough for `locus-shell` to consume, but generated by `locus-codegen` from the schema.

## Phase 5: Add Locus-Shell Graph Adapter

- Add `provider/locus` crate in `locus-shell` with package name `locus-provider`.
- Depend on `locus-dbus` by path initially:
  - `../locus/locus-dbus`
- Do not reimplement the D-Bus protocol.
- Wrap `locus-dbus` proxies into UI-friendly primitives:
  - connect to session bus
  - watch a typed `FieldBinding<T>`
  - decode updates into `T`
  - expose an async stream or callback channel
- Prefer `WatchNode` for property-bound UI fields, because it tracks both target changes and property changes.
- Keep D-Bus work off the GTK thread.
- Keep the separate `provider/dbus` crate focused on generic pure D-Bus object/property bindings such as UPower.

## Phase 6: Bridge Into Relm4

- Add a small runtime helper that forwards typed field updates into Relm4 input messages.
- Generated/macro code should maintain a `model.locus` submodel:
  - `model.locus.project_name: String`
  - `model.locus.workspace_id: u32`
  - `model.locus.workspace_focused: bool`
- Relm4 views stay simple:

```rust
gtk::Label {
    #[watch]
    set_label: model.locus.project_name.as_str(),
}
```

- Avoid `format!` in hot watched setters where possible. Precompute display strings in model state if needed.

## Phase 7: Redesign The Locus Macro Around Typed Bindings

Replace raw string tuple bindings:

```rust
project_name = ("context:selected", ["window", "workspace", "project"], "name")
```

with typed generated bindings:

```rust
project_name = paths::selected_project().property(Project::name)
workspace_id = paths::selected_workspace().property(Workspace::id)
```

The macro should:

- infer or validate the Rust field type from `FieldBinding<T>`
- generate `model.locus`
- subscribe on init
- emit Relm4 messages on updates
- update cached model state
- preserve the normal Relm4 component body

## Phase 8: Update Dev Widgets As Integration Tests

- Keep `dev-widgets` as the consumer proving ground.
- Replace hardcoded fake state with typed generated bindings once available.
- Add one primitive top strip bound to:
  - selected workspace id/name
  - selected project name
- Add one transient overlay bound to a simple boolean/string property.
- Keep all widget placement configs local to `dev-widgets`.

## Phase 9: Documentation And Migration

- Update `~/proj/locus/README.md` with typed property schema syntax.
- Update `locus-shell/PROJECT.md` and `PLAN.md` to say generated Rust contracts are the bridge between Locus and GTK.
- Document the layering:
  - YAML schema defines graph model
  - `locus-codegen` generates Rust contracts
  - `locus-dbus` owns protocol proxies
  - `locus-shell/provider/locus` adapts typed Locus graph bindings to provider updates
  - `locus-shell/provider/dbus` adapts generic D-Bus properties to provider updates
  - `locus-shell/provider/common` exposes feature-gated typed common D-Bus service definitions
  - `locus-shell/shell/macros` hides Relm4 boilerplate
- Keep raw string APIs available as escape hatches for dynamic or experimental schemas.

## Assumptions

- YAML remains the source format for now.
- D-Bus wire values remain strings initially.
- Generated Rust contracts are for known schemas at build/codegen time.
- Runtime arbitrary user schemas still require dynamic string-based fallbacks.
- `locus-shell` should not depend on `locus-core`; it should depend on `locus-dbus` and generated contract code.
